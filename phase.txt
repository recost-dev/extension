## Phase 1: Fingerprint Registry

### GitHub Issue

**Title:** Implement SDK Fingerprint Registry for provider/cost decoupling

**Labels:** `enhancement`, `architecture`, `phase-1`

**Description:**

#### Problem

Our scanner currently hardcodes provider detection logic, endpoint mappings, and pricing data inside individual matcher files (`openai-compatible.ts`, `provider-anthropic.ts`, `stripe.ts`, etc.). This creates several problems:

- Pricing updates require touching parser code
- Adding a new provider means writing a full matcher instead of just adding data
- There's no single source of truth — `endpoint-classification.ts` maintains its own `HOST_PROVIDER_MAP` of 50+ entries separately from the matchers
- Community contributions are hard because contributors need to understand the matcher internals just to add a provider

#### Solution

Create a **fingerprint registry** — a structured data layer that maps SDK method signatures to their HTTP endpoints, cost models, and pricing. The registry is a set of per-provider JSON files with a typed loader. Matchers continue doing detection via regex (for now) but delegate all classification and pricing to the registry.

#### Scope

- Design and implement the registry JSON schema and TypeScript types
- Populate JSON files for all currently supported providers (OpenAI, Anthropic, Stripe, Supabase, Firebase, AWS Bedrock, Gemini, Cohere)
- Build a registry loader with `lookupMethod()` and `lookupHost()` APIs
- Refactor all matcher files to pull pricing/endpoint data from the registry instead of hardcoding
- Migrate `HOST_PROVIDER_MAP` from `endpoint-classification.ts` into the registry
- Write unit tests for the registry
- Write `CONTRIBUTING.md` for community provider submissions

#### Acceptance Criteria

- [ ] All pricing constants removed from matcher files
- [ ] All matchers produce identical `ApiCallMatch` output as before (verified by tests)
- [ ] `HOST_PROVIDER_MAP` lives in one place (the registry)
- [ ] Adding a new provider requires only a new JSON file and no code changes
- [ ] `CONTRIBUTING.md` exists with clear schema docs and examples
- [ ] All existing tests pass

#### Branch

`feat/fingerprint-registry` off `main`

#### Risk

Low. Phases 1.1–1.4 are purely additive (new files only). Phase 1.5 refactors matchers one at a time — each can be tested independently. No detection logic changes.

#### Prompt 1.2 — Design the registry JSON schema

```
I'm building a fingerprint registry for my VS Code extension scanner (recost). 
The registry maps SDK method calls to their HTTP endpoints and cost data.

Based on these requirements, design:

1. A JSON schema for per-provider registry files (each provider gets its own .json file 
   in a fingerprints/ directory). The schema needs to support:
   - Provider name and display name
   - Language(s) supported (python, javascript, typescript, go, java, ruby, rust)
   - Package name(s) (npm package, pip package, etc.)
   - Method signature patterns (e.g., "client.chat.completions.create")
   - Mapped HTTP endpoint for each method (e.g., "POST /v1/chat/completions")
   - Cost model type per method: "per_token", "per_transaction", "per_request", "free"
   - Pricing fields: input_price_per_1m, output_price_per_1m, fixed_fee, percentage_fee
   - A top-level schema version field
   - Host patterns for the provider (e.g., "api.openai.com") — migrating from HOST_PROVIDER_MAP

2. TypeScript types in fingerprints/types.ts that match this schema exactly. Include:
   - ProviderFingerprint (the top-level type for a provider JSON file)
   - MethodFingerprint (individual method entry)
   - CostModel union type
   - HostPattern type
   - FingerprintRegistry (the loaded/indexed registry)

Make the schema extensible — it should be easy to add fields later without breaking 
existing files. Use optional fields where appropriate.

Create the types file. Don't create the JSON files yet — that's the next step.
```

#### Prompt 1.4 — Build the registry loader

```
I have a fingerprints/ directory with per-provider JSON files and TypeScript types. 
Now I need a loader.

Create fingerprints/registry.ts that:

1. Imports all provider JSON files at build time (they'll be bundled with the VS Code extension)
2. On initialization, indexes all methods by provider name + method pattern for O(1) lookup
3. Also indexes all host patterns for hostname-to-provider resolution
4. Exposes these functions:
   - lookupMethod(provider: string, methodChain: string): MethodFingerprint | null
     Given a provider name and a method chain like "client.chat.completions.create",
     returns the matching fingerprint entry or null
   - lookupHost(hostname: string): string | null
     Given a hostname like "api.openai.com", returns the provider name or null
   - getAllProviders(): string[]
     Returns all registered provider names
   - getProviderMethods(provider: string): MethodFingerprint[]
     Returns all methods for a given provider

5. Handle edge cases:
   - Method chains that don't include the variable name 
     (e.g., "chat.completions.create" should match even without the "client." prefix)
   - Host patterns with wildcards (e.g., "*.openai.com")
   - Case-insensitive provider name matching

Write the loader so it works in both Node.js (tests) and the VS Code extension 
webview context. No filesystem reads at runtime — everything is imported/bundled.

Include JSDoc comments on all public functions.
```

#### Prompt 1.7 — Write registry tests

```
Write comprehensive unit tests for the fingerprint registry system. Create a test file 
at fingerprints/__tests__/registry.test.ts (or whatever test convention the project uses).

Test categories:

1. lookupMethod() tests:
   - Known OpenAI method ("chat.completions.create") returns correct pricing
   - Known Anthropic method ("messages.create") returns correct pricing
   - Known Stripe method ("paymentIntents.create") returns correct cost model
   - Known Supabase method returns correct entry
   - Method with "client." prefix matches same as without
   - Unknown method returns null
   - Empty string returns null

   - "api.openai.com" returns "openai"
   - "api.anthropic.com" returns "anthropic"
   - "api.stripe.com" returns "stripe"
   - Wildcard patterns match correctly
   - Unknown hostname returns null
   - Empty string returns null

3. getAllProviders() tests:
   - Returns all expected providers
   - Returns no duplicates

4. getProviderMethods() tests:
   - Returns all methods for "openai"
   - Returns empty array for unknown provider

5. JSON validation tests:
   - Every JSON file in fingerprints/ parses successfully
   - Every JSON file conforms to the TypeScript ProviderFingerprint type
   - Every method entry has required fields (method pattern, endpoint, cost model)
   - No duplicate method patterns within a provider

6. Integration tests:
   - Run the full scanner on a sample file and verify ApiCallMatch objects 
     contain registry-sourced pricing (not hardcoded values)

Also run the full existing test suite and report any failures.
```
