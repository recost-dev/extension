# Phase 3: Waste Detection Rebuild (FULLY UPDATED)

**Title:** Rebuild waste detection on AST for structural accuracy

**Labels:** `enhancement`, `detection`, `phase-3`

**Description:**

### Problem

The current waste detector (`local-waste-detector.ts`) uses regex guards to detect 6 waste patterns (cache misses, unbatched loops, N+1 queries, rate limit risks, concurrency issues, redundant calls). These regex checks are:

1. **Unreliable** — regex can't distinguish a cache check from a comment mentioning caching
2. **Context-blind** — can't tell if a guard actually protects the specific API call or a different one
3. **Poorly calibrated** — the confidence scoring formula (0.48 base + score × 0.07) has no empirical basis

### Solution

Rebuild all waste detection on top of the AST (Phase 2). Instead of regex, we walk the tree structure to determine if an API call is inside a loop, behind a cache guard, missing a batch alternative, etc. Then recalibrate scoring using real-world data. Add cross-file resolution to catch API calls hidden behind wrappers, class methods, middleware, and barrel file re-exports.

### Scope

- Rebuild cache waste detection using AST
- Rebuild batch/N+1 detection using AST
- Rebuild rate limit and concurrency detection using AST
- Recalibrate confidence scoring with real repo data
- Cross-file import tracing covering utility wrappers, class services, middleware, re-export chains, and callback references

### Acceptance Criteria

- [ ] All 6 waste patterns detected with same or better recall than regex
- [ ] False positive rate measurably lower than regex approach
- [ ] Confidence scores calibrated against 3+ real repos
- [ ] Cross-file resolution catches wrapped API calls (utility functions, class methods, middleware)
- [ ] Cross-file resolution follows re-export chains up to 2 levels
- [ ] Old regex waste detection code removed (AST is the only path)

### Dependencies

- Phase 2 (Tree-sitter) must be merged first

### Branch

`feat/ast-waste-detection` off `feat/tree-sitter`

### Prompt 3.1 — Rebuild cache waste detection on AST

```
I'm rebuilding the waste detection in my scanner to use AST instead of regex.
Starting with cache waste detection.

The current regex approach in local-waste-detector.ts looks for "repeated reads, 
auth/config lookups, hot path, no cache guard" using text pattern matching. 
This produces false positives and misses real issues.

Create ast/waste/cache-detector.ts that:

1. Takes a list of API call matches (with their AST node references from Phase 2)
2. For each matched API call, checks:
   a. Is the call inside a function that's called from a request handler 
      (Express route, API handler, etc.)? → hot path
   b. Is the call inside a loop? (use the frequency analyzer from 2.7)
   c. In the same scope, is there a preceding variable assignment that looks like 
      a cache check? Patterns to detect:
      - const cached = cache.get(key); if (cached) return cached;
      - if (map.has(key)) return map.get(key);
      - if (memo[key]) return memo[key];
      - Redis/Memcached get before the API call
   d. Is the same API call (same method + same arguments) made multiple times 
      in the same scope? → redundant call
   e. Is it an auth/config call (e.g., fetching API keys, config values) 
      that could be done once at startup?

3. For each detected issue, produce a LocalWasteFinding with:
   - type: "cache"
   - description of what was found
   - severity: based on context (hot path + no guard = high, loop + no guard = high, 
     single scope redundancy = medium)
   - confidence: based on how certain the AST analysis is 
     (exact match = high, heuristic = medium)
   - suggestion: what the developer should do

4. Return LocalWasteFinding[] in the same format the existing detector uses

Write tests with sample code:
- API call in a loop with no cache → should detect
- API call in a loop with a cache check before it → should NOT detect
- Same API call twice in same function → should detect redundancy
- API call in a request handler with no memoization → should detect
- API call at module top level (one-time init) → should NOT detect
```

### Prompt 3.2 — Rebuild batch and N+1 detection on AST

```
Rebuild batch and N+1 waste detection using AST analysis.

Create ast/waste/batch-detector.ts that:

1. Detects API calls inside loops that could be batched:
   a. Walk each API call match's AST ancestors
   b. If the call is inside a for/while/forEach/map loop body:
      - Check the fingerprint registry for the matched method — does the provider 
        offer a batch alternative? (e.g., single create vs batch create)
      - If a batch alternative exists → flag as "batchable"
      - If no batch alternative but the call is making individual requests that 
        could be combined → flag as "potential-batch"

2. Detects N+1 patterns:
   a. A database/API query that returns a list
   b. Followed by a loop over that list
   c. Where each iteration makes another API call
   Pattern: results = await getList(); for (r of results) { await getDetail(r.id); }
   Detection: find a call expression whose result is used as the iterable in a 
   subsequent loop, where the loop body contains another API call

3. Detects sequential awaits that could be parallel:
   a. Multiple await statements in the same scope calling the same provider
   b. Where the calls are independent (result of one is not used as input to another)
   c. These could be wrapped in Promise.all()

4. For each finding, produce a LocalWasteFinding with:
   - type: "batch" or "n+1" or "sequential"
   - The specific API calls involved
   - severity and confidence
   - suggestion (use batch API, use Promise.all, restructure query)

Write tests:
- API call inside forEach → should detect batch opportunity
- API call inside loop with batch alternative in registry → should flag with suggestion
- Classic N+1: fetch list then loop-fetch details → should detect
- Two independent awaits to same provider → should suggest Promise.all
- Two dependent awaits (second uses first's result) → should NOT flag
- API call inside loop but it's a batch endpoint already → should NOT flag
```

### Prompt 3.3 — Rebuild rate limit and concurrency detection on AST

```
Rebuild rate limit risk and concurrency waste detection using AST.

Create ast/waste/concurrency-detector.ts that:

1. Detects polling without backoff:
   - API call inside setInterval() → "polling" risk
   - API call inside a while(true) with setTimeout → "polling" risk
   - Check if there's exponential backoff logic nearby 
     (multiplying delay, using a backoff library)
   - If no backoff → high severity

2. Detects unbounded concurrency:
   - Promise.all() wrapping an array.map() that makes API calls
   - No concurrency limiter (p-limit, p-queue, semaphore)
   - The array could be arbitrarily large
   - Flag: "unbounded parallel requests risk hitting rate limits"

3. Detects retry storms:
   - API call inside a catch block that re-calls the same function
   - No delay between retries
   - No max retry count
   - Flag: "retry without backoff can cause rate limit cascade"

4. Detects subscription/event amplification:
   - API call inside an event handler (addEventListener, .on(), .subscribe())
   - No debounce or throttle
   - Events that fire frequently (scroll, input, keydown, message, data)
   - Flag: "high-frequency event triggers API calls without debouncing"

5. For each finding:
   - type: "rate-limit", "concurrency", "retry-storm", "event-amplification"
   - severity based on risk level
   - confidence based on how clearly the AST shows the pattern
   - suggestion with specific fix

Write tests:
- fetch() inside setInterval with no backoff → detect
- fetch() inside setInterval with exponential backoff → skip
- Promise.all(items.map(item => api.call(item))) → detect unbounded
- Promise.all with p-limit wrapping → skip
- Recursive retry in catch with no delay → detect
- API call in scroll event handler with no debounce → detect
- API call in click handler → skip (low frequency, acceptable)
```

### Prompt 3.5 — Cross-file analysis (UPDATED)

```
This is the cross-file resolution phase. It catches API calls hidden behind wrappers, 
utility functions, class methods, middleware, and barrel file re-exports that the 
per-file scanner can't resolve on its own.

Common patterns this needs to catch:

Pattern 1 — Utility function wrapper:
  // lib/api.ts
  import OpenAI from "openai";
  const client = new OpenAI();
  export async function askGPT(prompt: string) {
    return client.chat.completions.create({ model: "gpt-4o", messages: [...] });
  }
  // routes/handler.ts  
  import { askGPT } from "../lib/api";
  for (const item of items) {
    await askGPT(item.question);  // ← API call in a loop
  }

Pattern 2 — Class-based service:
  // services/ai-service.ts
  import OpenAI from "openai";
  export class AIService {
    private client = new OpenAI();
    async complete(prompt: string) {
      return this.client.chat.completions.create({...});
    }
  }
  // routes/chat.ts
  import { AIService } from "../services/ai-service";
  const ai = new AIService();
  app.post("/chat", async (req, res) => {
    const result = await ai.complete(req.body.prompt);  // ← API call per request
  });

Pattern 3 — Middleware:
  // middleware/rate-check.ts
  import Stripe from "stripe";
  const stripe = new Stripe("sk_...");
  export function checkSubscription(req, res, next) {
    const sub = await stripe.subscriptions.retrieve(req.user.subId);
    req.subscription = sub;
    next();
  }
  // app.ts
  import { checkSubscription } from "./middleware/rate-check";
  app.use(checkSubscription);  // ← Stripe call on EVERY request

Pattern 4 — Re-export chain:
  // lib/openai.ts — defines askGPT with API call inside
  // lib/index.ts — export { askGPT } from "./openai"
  // routes/handler.ts — import { askGPT } from "../lib"
  // Scanner needs to follow: handler → lib/index → lib/openai → API call

Pattern 5 — Callback passed cross-file:
  // lib/processors.ts
  export async function processItem(item) {
    return openai.chat.completions.create({...});
  }
  // worker.ts
  import { processItem } from "./lib/processors";
  items.forEach(processItem);  // ← API call in loop via callback reference

Create ast/cross-file-resolver.ts that:

1. FIRST PASS — Build the export registry:
   Run after all per-file AST scans are complete. For each scanned file:
   
   a. Collect all exported functions (named exports, default exports):
      - If the function body contains a direct API call match (from per-file scan), 
        tag it: { file, exportName, apiCalls: ApiCallMatch[] }
   
   b. Collect all exported class declarations:
      - For each exported class, check which methods contain API calls 
        (use the class registry built in prompt 2.4)
      - Tag: { file, className, methods: { methodName: ApiCallMatch[] }[] }
   
   c. Collect all re-exports:
      - export { foo } from "./bar"  → record that foo originates from ./bar
      - export * from "./bar"        → record that all exports of ./bar are re-exported
      - Follow re-export chains up to 2 levels deep (covers barrel files like 
        lib/index.ts → lib/openai.ts)
   
   d. Collect middleware registrations that reference imported functions:
      - app.use(importedFunction) → record that importedFunction runs per-request
      - router.use(path, importedFunction) → same

2. SECOND PASS — Resolve imports and propagate:
   For each scanned file, look at its imports from local paths (relative imports only):
   
   a. Function call resolution:
      - If an import resolves to a tagged function in the export registry, 
        find every call site of that function in the importing file
      - Create a new ApiCallMatch at each call site, copying the API call metadata 
        from the export registry
      - Preserve the original call's context (loop, handler, etc.) for waste detection
   
   b. Class instance resolution:
      - If a file imports a class and creates an instance (new AIService()), 
        find all method calls on that instance
      - For each method call, check if that method is tagged in the export registry
      - If yes, create an ApiCallMatch at the call site
   
   c. Re-export resolution:
      - If an import path points to a file that re-exports from another file, 
        follow the chain (up to 2 levels) to find the original definition
      - Then apply rules (a) and (b) against the resolved definition
   
   d. Middleware resolution:
      - For middleware registrations collected in pass 1 that reference imported functions:
        resolve the import → check if the function is tagged → if yes, create an 
        ApiCallMatch tagged as frequency "per-request" since middleware runs on every hit
   
   e. Callback reference resolution:
      - If an imported function is passed by reference (not called) to .forEach, .map, 
        Promise.all, or a queue processor:
        resolve the import → check if the function is tagged → if yes, create an 
        ApiCallMatch and mark it with the appropriate loop/concurrency context

3. LIMITS (keep it tractable):
   - Re-export chains: follow max 2 levels deep
   - Class resolution: only direct instances (new Class()), not factory functions 
     that return instances
   - No dynamic imports or conditional requires
   - No resolution through node_modules (only relative imports)
   - No higher-order function returns (function that returns a function that calls API)
   - No decorator unwrapping
   - One level of function wrapping only (if askGPT calls helperThatCallsAPI 
     which calls openai, we only catch askGPT → openai, not the double-wrap 
     through helperThatCallsAPI unless helperThatCallsAPI was already tagged in pass 1)

4. INTEGRATION:
   - Run cross-file resolution after all per-file AST scans complete
   - Inject propagated matches into the main results
   - Deduplicate with existing matches (same file + line = keep the one with more data)
   - The waste detector then sees things like "askGPT in a loop" as 
     "OpenAI chat.completions.create in a loop" and flags it correctly

5. PERFORMANCE:
   - The first pass is O(files) — scan exports once per file
   - The second pass is O(files × imports) — for each file, resolve its imports
   - Cache parsed ASTs from the per-file scan so we don't re-parse
   - Measure total scan time with cross-file enabled vs disabled
   - If it adds more than 50% to scan time on a large repo (1000+ files), 
     add a config option to disable it

Write tests:

Pattern 1 test — Utility wrapper:
  Create two files: lib/api.ts (exports askGPT with OpenAI call inside) 
  and routes/handler.ts (imports askGPT, calls it in a loop)
  → Verify handler.ts shows an ApiCallMatch for OpenAI at the askGPT call site
  → Verify waste detector flags it as API-call-in-loop

Pattern 2 test — Class service:
  Create services/ai.ts (exports AIService class with complete() method calling OpenAI)
  and routes/chat.ts (imports, instantiates, calls ai.complete())
  → Verify chat.ts shows an ApiCallMatch for OpenAI at ai.complete()

Pattern 3 test — Middleware:
  Create middleware/auth.ts (exports checkSub that calls Stripe)
  and app.ts (app.use(checkSub))
  → Verify app.ts shows an ApiCallMatch for Stripe tagged as per-request

Pattern 4 test — Re-export chain:
  Create lib/openai.ts (exports askGPT), lib/index.ts (re-exports from openai.ts),
  and routes/handler.ts (imports from lib/index)
  → Verify handler.ts resolves through the barrel to the original API call

Pattern 5 test — Callback reference:
  Create lib/processors.ts (exports processItem with OpenAI call)
  and worker.ts (imports processItem, passes to items.forEach(processItem))
  → Verify worker.ts shows API-call-in-loop

Edge case tests:
- Import from node_modules (should NOT be resolved cross-file)
- Re-export chain 3 levels deep (should stop at 2, use what it has)
- Class factory function returning new AIService() (should NOT resolve — known limit)
- File with no exports (should be skipped in first pass)
- Circular imports between two files (should not infinite loop — detect and skip)
```
