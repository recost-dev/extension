# Arbitrary File Write via Unvalidated Path in handleApplyFix/openFile (VS Code Extension)

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-04-03 04:36:27 UTC
**Target:** https://github.com/ecoapi-dev/extension
**Endpoint:** src/webview-provider.ts (handleApplyFix, handleOpenFile)
**Method:** postMessage → host handlers
**CWE:** CWE-22
**CVSS:** 8.1

## Description

The extension trusts file paths supplied from its webview and uses them to open and modify files via the host API without validating confinement to the active workspace.

Root cause
- The webview sends messages containing a user/remote-controlled file path (e.g., from remote “suggestions” data or AI chat results) through the postMessage channel.
- The extension’s host-side handlers handleApplyFix and handleOpenFile receive message.file and construct a file URI with:
  const fileUri = vscode.Uri.joinPath(workspaceFolder.uri, file);
  This approach does not prevent traversal: segments like ../ are normalized, allowing resolution outside the workspace root.
- The handlers then open and edit the target document, persisting changes with doc.save().

Impact
- A malicious suggestion or crafted UI flow can cause a user click on “Apply fix” or “Open file” to read or write arbitrary files on the user’s machine that are accessible to the VS Code process, including files outside the workspace. This enables exfiltration or tampering with sensitive local files.

Evidence
- In src/webview-provider.ts:
  - handleApplyFix (lines ~1840-1869) builds fileUri from message.file with vscode.Uri.joinPath and immediately edits/saves the file.
  - handleOpenFile (lines ~1971-1987) uses the same joinPath pattern, enabling outside-of-workspace opens.
- In webview/src/components/ResultsPage.tsx, the file path used for applyFix can originate from server-returned “suggestions” (affectedFiles[]), which are merged into the UI with limited normalization. If no local match is found, the code falls back to suggestion.affectedFiles[0], allowing remote influence over message.file.

This combination creates an arbitrary file write/read risk triggered by a trusted UI action but sourced from remote/untrusted data.

## Impact

Exploitation enables arbitrary file read/write in the context of the user running VS Code:
- Write to or modify shell profiles (e.g., ~/.bashrc, ~/.zshrc) to achieve persistence
- Overwrite local configuration files (e.g., IDE settings, project config) to implant malicious behavior
- Tamper with source files outside the intended workspace boundary
- Read sensitive local files by opening them in the editor (e.g., SSH keys, API keys, credentials) and potentially leaking through subsequent operations

Business impact includes risk of developer workstation compromise, data leakage, and supply-chain integrity issues if tampered files are later committed or executed.

## Technical Analysis

The vulnerability results from insufficient validation of a user-controlled path parameter crossing the trust boundary from a webview into the extension host.

Key flows:
1) Webview event to host (postMessage)
   - ResultsPage.tsx’s “Apply fix” button posts { type: "applyFix", code, file, line }.
   - The file is derived via resolveSuggestionTarget() which can fall back to suggestion.affectedFiles[0]; the latter may be sourced from remote backend responses (getAllSuggestions) merged into the UI, not guaranteed to be confined to the project.

2) Host-side handling (webview-provider.ts)
   - handleApplyFix(message.code, message.file, message.line):
       const fileUri = vscode.Uri.joinPath(workspaceFolder.uri, file);
       const doc = await vscode.workspace.openTextDocument(fileUri);
       ...
       await editor.edit(...);
       await doc.save();
   - handleOpenFile(message.file, message.line) uses the same joinPath pattern and then opens the file.

Using joinPath with untrusted relative paths enables traversal. Example:
- If workspaceFolder.uri.fsPath is /home/user/work/project,
  vscode.Uri.joinPath(..., "../../.bashrc") normalizes to /home/user/.bashrc, escaping the workspace.

No checks enforce that the resolved path remains within the workspace root; absolute paths are not rejected. Therefore, a remote suggestion can direct the extension to write to arbitrary filesystem locations (subject to OS permissions) once the user clicks “Apply fix”.

Additional context:
- readWorkspaceFileExcerpt also uses joinPath for relativePath, but it is not directly driven from the untrusted webview message type.
- The LocalServer binds to 127.0.0.1 only (server.listen(0, "127.0.0.1", ...)), so it’s not directly exposed externally.
- Webview CSP limits scripts to nonce-based local files; the attack surface here is specifically the host API path handling.

## Remediation

Implement strict workspace confinement and input validation before opening or modifying any file based on a webview-supplied path:
1) Reject absolute paths and any path containing null bytes.
2) Resolve the target using the workspace root with path.resolve(root, file) and compute a safe relative path:
   - rel = path.relative(root, resolved)
   - if rel starts with ".." or is absolute, reject.
3) Use vscode.Uri.file(resolved) once validated.
4) Apply the same checks in both handleApplyFix and handleOpenFile.
5) Consider normalizing all file paths carried in suggestions/endpoints to a workspace-relative, canonical form at ingestion time. If a suggestion file does not map to a known workspace file, display an error or require the user to choose a valid file.
6) Add unit tests to ensure traversal attempts are rejected (../../, symlinks if applicable).
7) Document the trust boundary: anything arriving from the webview must be validated and treated as untrusted input.

Example safe pattern (TypeScript):
- Validate: no NUL, not absolute
- resolved = path.resolve(workspaceRoot, file)
- rel = path.relative(workspaceRoot, resolved)
- if (rel.startsWith("..") || path.isAbsolute(rel)) { reject }

Adopt this uniformly anywhere the extension uses webview-provided file paths.
